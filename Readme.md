# Отчёт о горном походе — локальная сборка и автоматический CI

Статус сборки:
![Build status](https://github.com/Zararest/trekking-report/actions/workflows/build.yml/badge.svg)

Данный репозиторий содержит LaTeX-шаблон для отчёта о горном походе.
Сборка выполняется в изолированном Docker-контейнере, что гарантирует одинаковый результат на любой ОС.
Кроме того, настроена автоматическая сборка в **GitHub Actions** при каждом пуше.

---

## 📋 Требования

- **Docker** (версия 20.10 или новее)  
  Установка: [docker.com/get-started](https://www.docker.com/get-started)
- Docker должен быть **запущен** перед выполнением скриптов.

### Проверка Docker
- **Linux/macOS:** выполните `docker info` в терминале.
- **Windows:** откройте PowerShell и выполните `docker info`.
Если команда не найдена или выдаёт ошибку, Docker не установлен/не запущен.

---

## 🚀 Локальная сборка

### Linux / macOS

1. Откройте терминал в корневой папке проекта.
2. Дайте права на выполнение скрипта (один раз):
   ```bash
   chmod +x build.sh
   ```
3. Запустите сборку:
   ```bash
   ./build.sh
   ```
   Для режима наблюдения (автоматическая пересборка при изменениях):
   ```bash
   ./build.sh --watch
   ```
4. Готовый PDF появится в той же папке (`main.pdf`).

### Windows (PowerShell)

1. Откройте PowerShell в корневой папке проекта.
2. Выполните скрипт:
   ```powershell
   .\build.ps1
   ```
   Для режима наблюдения:
   ```powershell
   .\build.ps1 -watch
   ```
3. При первом запуске Docker скачает образ (около 2–3 ГБ).
4. После компиляции файл `main.pdf` будет создан в текущей папке.

---

## 🔄 Режим наблюдения (--watch / -watch)

При активной работе над отчётом вы можете использовать режим наблюдения.
В этом режиме контейнер остаётся запущенным, а `latexmk` автоматически перекомпилирует документ каждый раз, когда вы сохраняете изменения в `.tex`-файлах.

Для остановки нажмите `Ctrl+C`. Контейнер будет удалён автоматически (флаг `--rm`), а все временные файлы останутся на хосте.

Этот режим идеален для финальной вёрстки, когда вы правите текст и хотите видеть изменения мгновенно.

## 🤖 Автоматическая сборка в GitHub CI

При каждом пуше в репозиторий (в любую ветку) или создании pull request запускается **GitHub Actions workflow**, который:

1. Загружает ваш код.
2. Компилирует `main.tex` в PDF с помощью того же Docker-образа, что и при локальной сборке.
3. Сохраняет готовый PDF как **артефакт** сборки.

### Где скачать готовый PDF

1. Перейдите на страницу вашего репозитория на GitHub.
2. Нажмите вкладку **Actions**.
3. Выберите нужный запуск (обычно последний).
4. В разделе **Artifacts** нажмите `tour-report` – скачается ZIP-архив с файлом `main.pdf`.

### Бейдж статуса сборки

В начале этого файла находится бейдж, показывающий состояние последней сборки.
- ✅ Зелёный – сборка успешна.
- ❌ Красный – произошла ошибка (подробности можно посмотреть во вкладке Actions).

Чтобы бейдж работал, замените `USERNAME/REPO` в ссылке на свои данные.

## 🔧 Что делают скрипты

- Проверяют, запущен ли Docker.
- Определяют архитектуру процессора и выбирают подходящий образ (`zydou/texlive:latest` для ARM, `ghcr.io/xu-cheng/texlive-full:20250301` для x86_64).
- Загружают образ (один раз, потом кэшируется).
- Монтируют текущую папку внутрь контейнера как `/workspace`.
- Запускают `latexmk` с флагами `-pdf -file-line-error -halt-on-error main.tex`.
- При успехе сохраняют `main.pdf` на хосте.

---

## 📦 Используемые Docker-образы

- **Для Intel/AMD:** `ghcr.io/xu-cheng/texlive-full:20250301` – полный TeXLive, фиксированная версия.
- **Для ARM64:** `zydou/texlive:latest` – полный TeXLive, multi-arch образ.

В CI используется `xu-cheng/latex-action`, который по умолчанию берёт `ghcr.io/xu-cheng/texlive-full:latest`, но вы можете переопределить.

## 🐛 Возможные проблемы и их решение

- **Ошибка «docker: command not found»** — Docker не установлен или не добавлен в PATH.
- **Ошибка «Cannot connect to the Docker daemon»** — Docker не запущен. Запустите Docker Desktop (Windows/macOS) или службу Docker (Linux).
- **Ошибка «no matching manifest for linux/arm64»** — автоматически решается скриптом (используется `zydou/texlive`). Если всё же возникает, проверьте обновлённость Docker Desktop.
- **Ошибка доступа к папке** — на Windows убедитесь, что диск смонтирован в настройках Docker Desktop (Settings → Resources → File Sharing).
- **Отсутствуют шрифты** — если документ использует системные шрифты, их нужно смонтировать дополнительно. Обычно для стандартных шрифтов TeXLive этого не требуется.
